# v0.9 Real Project Proof Plan

This plan defines the next validation milestone before Game Dev Studio can be considered for `v1.0.0`.

It is a validation plan, not a runtime Skill reference. Keep it out of `SKILL.md` reference loading so normal tasks remain token-conscious.

## Goal

Use Game Dev Studio on one independently scoped game project and produce a proof report that can be checked by `scripts/validate-proof-package.mjs`.

The purpose is to prove the Skill can move from client request to implementation delivery without becoming only a code helper, accepting placeholder visuals, or collapsing architecture boundaries.

## Candidate Project Criteria

Use a project that is small enough to finish but real enough to expose workflow problems.

Acceptable candidates:

- A small public web game with no required account setup
- A simple Godot sample project
- A small Unity sample with a documented open license
- A user-owned prototype with a clean before/after commit

Avoid candidates that require:

- Private credentials
- Large binary assets
- Paid store SDKs
- Long package installs
- Legal, contract, pricing, or payment workflow
- Tuning the Skill toward one specific game

## Proof Workflow

1. Record the source repository and starting commit.
2. Use a client-style prompt that asks the studio to produce a bounded first playable or polished slice.
3. Follow Client Studio Production Workflow:
   - brief intake
   - studio proposal
   - scope lock
   - formal art direction or style bible
   - production architecture gate
   - production task breakdown
4. Move into Studio Implementation Delivery Workflow only after scope and architecture are clear.
5. Inspect the existing repository before editing.
6. Implement a small bounded slice.
7. Run available build, boot, smoke, or unit checks.
8. Capture screenshot or visual QA evidence when UI or visuals change.
9. Write QA, Ruthless Playtester, and client acceptance notes.
10. Create `validation/proof-<project>-v1.0.0.md`.
11. Run `node scripts/validate-proof-package.mjs --require validation/proof-<project>-v1.0.0.md`.

## Evidence Rules

- Do not commit copied third-party project source into this Skill repository.
- Do not treat planning-only output as implementation proof.
- Do not mark visual QA as `Passed` without screenshot or visual comparison evidence.
- Do not mark tests as passed unless the command actually ran.
- If browser, engine, or package setup is blocked, record `Blocked` with the reason.
- Keep project-specific rules inside the proof report; do not move them into general Skill rules.

## v0.9 Exit Criteria

`v0.9.0` is ready only when:

- At least one proof report exists.
- The proof report passes `scripts/validate-proof-package.mjs --require`.
- The report proves implementation, not only planning.
- The report names real tests and their results.
- The report records visual evidence or an honest blocked status.
- The proof does not make the Skill specific to the selected project.

`v0.9.0` should not be promoted to `v1.0.0` until the proof package is reviewed and the remaining risks are resolved or explicitly accepted.
