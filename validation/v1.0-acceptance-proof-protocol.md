# v1.0 Acceptance Proof Protocol

This protocol defines the evidence required before Game Dev Studio can be treated as a stable client-commissioned AI game development studio operating system.

It is a validation artifact, not a runtime reference. Do not load it during normal Skill use unless the task is a release, maturity, or completion audit.

## Purpose

Game Dev Studio should not reach `v1.0.0` only because the documentation is complete.

The `v1.0.0` claim requires proof that the Skill can guide at least one independently scoped game project from client request to delivery without losing the studio workflow, visual quality bar, architecture boundaries, QA evidence, or client acceptance discipline.

## Required Proof Package

Before `v1.0.0`, create at least one complete proof package with:

- Source project name and repository URL or local fixture path
- Source commit before work begins
- Final commit after work ends
- Original client-style prompt
- References loaded by the Skill
- Client brief intake
- Studio proposal with two to four directions when the request is broad
- Scope lock with MVP, deferred items, non-goals, risks, and acceptance criteria
- Formal art direction or approved style bible
- Technical architecture gate with module boundaries
- Production task breakdown
- Implementation delivery plan
- Files inspected before editing
- Files changed
- QA commands and results
- Screenshot or visual QA evidence when UI or visuals changed
- Ruthless Playtester note
- Client acceptance report
- Remaining risks and client decisions

## Minimum External Proof

The strongest proof should use an existing game project that was not created only to make the validation pass.

Acceptable examples:

- A small public web game repository
- A public Godot demo project
- A small Unity or Unreal sample project
- A user-owned prototype with a clean before/after commit

A local fixture is useful as a smoke test, but it is not enough for final `v1.0.0` proof by itself.

## Evidence Quality Rules

- Mark tests as `Passed`, `Failed`, `Not run`, or `Blocked`.
- Do not count a screenshot requirement as passed if no screenshot or visual comparison was captured.
- Do not count a client workflow as proven if it only produced a plan and no implementation delivery.
- Do not count implementation as client-ready if the UI still uses placeholder rectangles, debug panels, default typography, or unapproved art.
- Do not count architecture as proven if gameplay, UI, render, VFX, audio, save, and data responsibilities collapse into one file.
- Do not treat prompt tests as runtime proof.

## v1.0 Release Gate

Only recommend `v1.0.0` when:

- The proof package exists.
- The proof package includes implementation, not only planning.
- The project runs or clearly records why runtime checks are blocked.
- Player-facing visual changes have visual evidence.
- The delivery report compares results against acceptance criteria.
- The Skill remains general-purpose and is not tuned to the proof project only.
- `validation/runtime-visual-qa-gate.md` is closed with at least one external proof screenshot or equivalent visual comparison artifact.
